# Проектирование производительности: кеширование и rate limiting

## 1. Анализ производительности

### Hot paths (частые операции)

| Endpoint                       | Частота | Обоснование                                                                           |
|--------------------------------|---------|---------------------------------------------------------------------------------------|
| `GET /v1/users?login=...`      | Высокая | Вызывается delivery-service и auth-service при каждом создании доставки и регистрации |
| `GET /v1/packages?user_id=...` | Высокая | Пользователи часто просматривают список своих посылок                                 |
| `POST /v1/auth/login`          | Высокая | Точка входа для всех пользователей; частая цель атак                                  |

### Медленные операции

| Операция                                      | Причина медлительности                             |
|-----------------------------------------------|----------------------------------------------------|
| `aggregate` в MongoDB (посылки пользователя)  | Полный скан коллекции для фильтрации по `user_id`  |

### Требования к производительности

- **Время отклика**: 95-й перцентиль ≤ 50 мс для `GET` операций
- **Пропускная способность**: ≥ 1000 RPS для чтения, ≥ 100 RPS для записи
- **Rate limit для login**: 10 запросов/минуту на IP (защита от brute-force)

## 2. Стратегия кеширования

### Выбранные данные для кеширования

#### `GET /v1/users?login=...` -> user-service

**Стратегия**: Cache-Aside (Lazy Loading)

**Обоснование**: Профили пользователей редко изменяются. Поиск по логину — самая частая операция (используется delivery-service, auth-service). Данные читаются намного чаще, чем пишутся.

**Ключ Redis**: `user:login:{login}`

**TTL**: 300 секунд (5 минут)

**Инвалидация**: При `POST /v1/users` (создание пользователя) — явное удаление ключа через `DEL`.

> **Примечание об инвалидации.** В текущем API отсутствует endpoint обновления профиля пользователя, а логин уникален — создать двух пользователей с одинаковым логином невозможно. Поэтому инвалидация при `POST /v1/users` носит превентивный характер (защита от race condition при параллельных запросах). В полноценной системе с endpoint'ом `PATCH /v1/users/{id}` инвалидация выполнялась бы именно там — при каждом изменении профиля.

---

#### `GET /v1/packages?user_id=...` -> package-service

**Стратегия**: Cache-Aside (Lazy Loading)

**Обоснование**: Список посылок пользователя читается чаще, чем пополняется. MongoDB-запрос относительно дорогой (агрегация по `user_id`). При создании новой посылки старый кеш моментально устаревает — нужна явная инвалидация.

**Ключ Redis**: `packages:user:{user_id}`

**TTL**: 120 секунд (2 минуты) — меньше, т.к. данные изменяются чаще

**Инвалидация**: При `POST /v1/packages` (создание посылки) — явное удаление ключа через `DEL`.

> **Примечание об инвалидации.** Посылки иммутабельны: в текущем API нет endpoint'ов редактирования или удаления. Единственное событие, изменяющее список посылок пользователя — создание новой посылки, поэтому инвалидация привязана к `POST /v1/packages`. В системе с endpoint'ами `PATCH /v1/packages/{id}` или `DELETE /v1/packages/{id}` инвалидация выполнялась бы и в этих обработчиках.

---

### Почему Cache-Aside, а не другие стратегии

| Стратегия         | Почему не выбрана                                                                                   |
|-------------------|-----------------------------------------------------------------------------------------------------|
| **Read-Through**  | Требует прокси-слоя между приложением и БД; усложняет архитектуру                                   |
| **Write-Through** | При создании пользователя/посылки нет смысла сразу писать в кеш — данные могут не быть востребованы |
| **Write-Back**    | Риск потери данных при падении Redis; неприемлемо для бизнес-данных                                 |

## 3. Стратегия rate limiting

### Выбранный endpoint: `POST /v1/auth/login`

**Обоснование**: Это единственная публичная точка входа без JWT-аутентификации, напрямую уязвимая к brute-force и credential stuffing атакам.

### Алгоритм: Fixed Window Counter

**Ключ Redis**: `rate_limit:login:{client_ip}`

**Лимит**: 10 запросов / 60 секунд на IP

**Реализация через Redis**:
```
count = INCR rate_limit:login:{ip}
if count == 1: EXPIRE rate_limit:login:{ip} 60
if count > 10: → 429 Too Many Requests
```

Атомарность `INCR` гарантирует корректность счётчика без race conditions.

### Почему Fixed Window, а не другие алгоритмы

| Алгоритм                   | Сравнение                                                                                                 |
|----------------------------|-----------------------------------------------------------------------------------------------------------|
| **Fixed Window Counter**   | Прост, атомарен (INCR), дёшев по памяти (1 ключ/IP). Граничный эффект неопасен для brute-force protection |
| **Sliding Window Log**     | Точнее, но хранит timestamp каждого запроса — дорого по памяти при высоком RPS                            |
| **Sliding Window Counter** | Чуть точнее Fixed Window, но требует 2 ключа и арифметики                                                 |
| **Token Bucket**           | Подходит для burst-трафика; для login нежелателен — burst атаки должны блокироваться жёстко               |
| **Leaking Bucket**         | Queue-based, избыточен для простого rate limiting                                                         |

### HTTP заголовки ответа

Возвращаются при каждом запросе к `/v1/auth/login`:

| Заголовок               | Значение | Описание                  |
|-------------------------|----------|---------------------------|
| `X-RateLimit-Limit`     | `10`     | Максимум запросов в окне  |
| `X-RateLimit-Remaining` | `0..10`  | Оставшиеся запросы        |
| `X-RateLimit-Reset`     | `60`     | Размер окна в секундах    |

При превышении лимита: `HTTP 429 Too Many Requests`.

## 4. Анализ влияния на производительность

### Кеширование

| Метрика                             | Без кеша                | С кешем (при hit rate 80%) |
|-------------------------------------|-------------------------|----------------------------|
| Latency `GET /v1/users?login=`      | ~10–20 мс (PostgreSQL)  | ~1–2 мс (Redis)            |
| Latency `GET /v1/packages?user_id=` | ~15–30 мс (MongoDB)     | ~1–2 мс (Redis)            |
| Нагрузка на PostgreSQL              | 100% запросов           | ~20% запросов              |
| Нагрузка на MongoDB                 | 100% запросов           | ~20% запросов              |

### Rate limiting

- Защита auth-service от перегрузки при атаках
- Снижение нагрузки на PostgreSQL (credential lookup) при DDoS/brute-force
- Latency проверки: < 1 мс (Redis INCR)
