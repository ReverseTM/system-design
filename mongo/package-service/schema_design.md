# Проектирование документной модели: package-service

## Коллекции

В package-service одна коллекция — `packages`.

---

## Структура документа

```json
{
  "_id": ObjectId,
  "user_id": NumberLong,
  "weight": Double,
  "dimensions": {
    "length": Double,
    "width": Double,
    "height": Double
  },
  "description": String,
  "created_at": Date
}
```

---

## Выбор: embedded vs references

### `dimensions` — **embedded document**

**Обоснование:**
- Габариты (длина, ширина, высота) являются неотделимой частью посылки: без них документ теряет смысл.
- Все три поля всегда читаются и записываются вместе — нет сценария, где нужна только ширина без длины.
- Выделение в отдельную коллекцию `dimensions` создало бы избыточный JOIN при каждом чтении без какой-либо выгоды.
- Количество полей фиксировано и мало (3 числа) — нет риска роста документа.

**Альтернатива (references):** не имеет смысла. Габариты не имеют независимого жизненного цикла.

---

### `user_id` — **reference (ссылка)**

**Обоснование:**
- Пользователь — сущность из отдельного микросервиса (`user-service`).
- Встраивать данные пользователя (имя, email и т.д.) нельзя: они могут меняться, а обновлять их во всех миллионах посылок — неприемлемо.
- В MongoDB cross-collection join нет транзакционной семантики, поэтому ссылка (`user_id: NumberLong`) — правильный выбор.
- Сервис намеренно хранит только `user_id`, а не профиль пользователя: это граница ответственности микросервиса.

**Альтернатива (embedded user):** нарушила бы принцип единой ответственности и привела к рассинхронизации данных.

---

## Индексы

```js
db.packages.createIndex(
  { user_id: 1, created_at: -1 },
  { name: "idx_packages_user_created" }
)
```

**Обоснование:** основной паттерн чтения — "все посылки пользователя, отсортированные по убыванию даты". Составной индекс покрывает фильтр по `user_id` и сортировку по `created_at` одновременно.
